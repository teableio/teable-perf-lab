# table-delete/10k-20f

## Goal

Measure repeated archive-to-trash requests for 10 independent 10,000-record
mixed 20-field tables in one run.

## Seed Phase

Create 10 deterministic tables in the seeded base. Each table has 10,000
records and 20 mixed fields, reusing the shared mixed-record seed from the
undo/redo family (`undoRedo10kBaseConfig`). Values are generated from the row
number so V1, V2, and repeated runs compare the same data.

The runner validates every seed table with a paged full row-count scan before
measurement. Each sample has its own seed-cache identity so repeated archive
requests never mutate the same table twice.

## Execute Phase

1. Reuse or create 10 seed tables.
2. For each sample, call `DELETE /api/base/{baseId}/table/{tableId}` (the
   archive-to-trash UI route, not the permanent variant) and capture
   `x-teable-v2*` routing headers.
3. Verify each table is gone from the base table list and that a matching item
   exists in `GET /api/trash/items?resourceType=base&resourceId={baseId}`.
4. Local cleanup restores every archived table through
   `POST /api/trash/restore/{trashId}` and full-scans 10,000 rows per table so
   the cached seeds stay reusable; if a restore fails, that table is
   permanently deleted so a later run reseeds instead of reusing a corrupted
   fixture.

## Primary Metric

- `deleteTableP95Ms`: historical p95 threshold key for the 10 measured archive
  `DELETE` requests. With 10 samples, the current nearest-rank percentile math
  makes this gate effectively the slowest request (max). V1 soft deletes table,
  field, and view metadata in one transaction after detaching links.

## Verification Metrics

- `deleteTableMinMs` / `deleteTableP50Ms` / `deleteTableMaxMs`: per-request
  distribution (diagnostic). `deleteTableMaxMs` is expected to match
  `deleteTableP95Ms` while the case keeps 10 samples.
- `deleteTableTotalMs`: sum of the 10 measured delete requests (diagnostic).
- `verifyMs`: table-list absence checks plus trash item lookups. Diagnostic
  only; it does not participate in the primary threshold.

## Notes

- Seed creation and validation are setup phases and stay out of the primary
  metric.
- The seed table has no link fields, so the `detachLink` pre-step inside the
  delete is cheap; a linked-table variant is a candidate follow-up case.
- Cleanup restore runs under per-sample `cleanupRestoreTable-*` trace steps and
  is never measured.
- Data-scaling companion: `table-delete/10k-20f-link-detach` covers the
  O(records) path where v1 `detachLink` converts a surviving link field
  during the delete.
