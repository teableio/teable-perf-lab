# table-delete/10k-20f

## Goal

Measure archiving (deleting to trash) a 10,000-record mixed 20-field table.

## Seed Phase

Create a deterministic table in the seeded base with 10,000 records and 20
mixed fields, reusing the shared mixed-record seed from the undo/redo family
(`undoRedo10kBaseConfig`). Values are generated from the row number so V1, V2,
and repeated runs compare the same data.

The runner validates the seed with a paged full row-count scan before
measurement.

## Execute Phase

1. Reuse or create the seed table.
2. Call `DELETE /api/base/{baseId}/table/{tableId}` (the archive-to-trash UI
   route, not the permanent variant) and capture `x-teable-v2*` routing
   headers.
3. Verify the table is gone from the base table list and that a matching item
   exists in `GET /api/trash/items?resourceType=base&resourceId={baseId}`.
4. Local cleanup restores the table through `POST /api/trash/restore/{trashId}`
   so the cached seed stays reusable; if the restore fails, the table is
   permanently deleted so a later run reseeds instead of reusing a corrupted
   fixture.

## Primary Metric

- `deleteTableRequestMs`: the archive `DELETE` request duration. V1 soft
  deletes table, field, and view metadata in one transaction after detaching
  links.

## Verification Metrics

- `deleteTableVerifyMs`: table-list absence check plus trash item lookup.
  Diagnostic only; it does not participate in the primary threshold.

## Notes

- Seed creation and validation are setup phases and stay out of the primary
  metric.
- The seed table has no link fields, so the `detachLink` pre-step inside the
  delete is cheap; a linked-table variant is a candidate follow-up case.
- Cleanup restore runs under the `cleanupRestoreTable` trace step and is never
  measured.
