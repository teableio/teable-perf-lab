# table-restore/10k-20f

## Goal

Measure repeated restore requests for 10 independent 10,000-record mixed
20-field tables from the base trash in one run.

## Seed Phase

Create 10 deterministic tables in the seeded base. Each table has 10,000
records and 20 mixed fields, reusing the shared mixed-record seed from the
undo/redo family (`undoRedo10kBaseConfig`). Values are generated from the row
number so V1, V2, and repeated runs compare the same data.

The runner validates every seed table with a paged full row-count scan before
measurement. Each sample has its own seed-cache identity so repeated restore
requests never reuse the same trash item.

## Execute Phase

1. Reuse or create 10 seed tables.
2. Setup per sample (not measured, `deleteSetup-*` steps): archive each table
   via `DELETE /api/base/{baseId}/table/{tableId}`, confirm it left the base
   table list, and resolve its `trashId` from
   `GET /api/trash/items?resourceType=base&resourceId={baseId}`.
3. Measured: run 10 `POST /api/trash/restore/{trashId}` requests, one per
   sample. Each response carries `x-teable-v2` /
   `x-teable-v2-feature: restoreTable` / `x-teable-v2-reason` headers recorded
   for routing evidence, and V1/V2 runs fail if the response did not use the
   requested engine.
4. Verify every restored table: a paged full scan proves all 10,000 rows are
   readable again, and sample rows (offsets 0 / 4,999 / 9,999) prove the
   `Title` and `External ID` text values survived the delete/restore round
   trip.
5. A successful run leaves all seed tables restored and reusable; on failure
   cleanup retries any still-archived table once and otherwise permanently
   deletes the fixture.

## Primary Metric

- `restoreTableP95Ms`: historical p95 threshold key for the 10 measured restore
  `POST` requests. With 10 samples, the current nearest-rank percentile math
  makes this gate effectively the slowest request (max).

## Verification Metrics

- `restoreTableMinMs` / `restoreTableP50Ms` / `restoreTableMaxMs`:
  per-request distribution (diagnostic). `restoreTableMaxMs` is expected to
  match `restoreTableP95Ms` while the case keeps 10 samples.
- `restoreTableTotalMs`: sum of the 10 measured restore requests
  (diagnostic).
- `setupMs`: archive-to-trash setup duration across all samples
  (diagnostic).
- `verifyMs`: full row-count scans plus sample text value checks. Diagnostic
  only; it does not participate in the primary threshold.

## Notes

- The trash item id is taken from `/api/trash/items`; operation-prefixed ids
  would skip the v2 routing decision, so the runner never uses them.
- Sample value checks are limited to plain text fields to stay format-stable
  across engines; row presence is proven by the full scan.
- Restore is a metadata flip today (~22 rows of table/field/view metadata),
  so this latency is record-count independent by design. The data-scaling
  sentinel lives in `table-restore/10k-20f-link-1k`, whose fixture owns
  10,000 populated link cells.
