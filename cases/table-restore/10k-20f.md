# table-restore/10k-20f

## Goal

Measure restoring a 10,000-record mixed 20-field table from the base trash.

## Seed Phase

Create a deterministic table in the seeded base with 10,000 records and 20
mixed fields, reusing the shared mixed-record seed from the undo/redo family
(`undoRedo10kBaseConfig`). Values are generated from the row number so V1, V2,
and repeated runs compare the same data.

The runner validates the seed with a paged full row-count scan before
measurement.

## Execute Phase

1. Reuse or create the seed table.
2. Setup (not measured, `deleteSetup` step): archive the table via
   `DELETE /api/base/{baseId}/table/{tableId}`, confirm it left the base table
   list, and resolve its `trashId` from
   `GET /api/trash/items?resourceType=base&resourceId={baseId}`.
3. Measured: `POST /api/trash/restore/{trashId}`. The response carries
   `x-teable-v2` / `x-teable-v2-feature: restoreTable` / `x-teable-v2-reason`
   headers recorded for routing evidence.
4. Verify: a paged full scan proves all 10,000 rows are readable again, and
   sample rows (offsets 0 / 4,999 / 9,999) prove the `Title` and `External ID`
   text values survived the delete/restore round trip.
5. A successful run leaves the seed table restored and reusable; on failure the
   cleanup retries the restore once and otherwise permanently deletes the
   fixture.

## Primary Metric

- `restoreTableRequestMs`: the restore `POST` request duration.

## Verification Metrics

- `deleteSetupMs`: archive-to-trash setup duration (diagnostic).
- `restoreTableVerifyMs`: full row-count scan plus sample text value checks.
  Diagnostic only; it does not participate in the primary threshold.

## Notes

- The trash item id is taken from `/api/trash/items`; operation-prefixed ids
  would skip the v2 routing decision, so the runner never uses them.
- Sample value checks are limited to plain text fields to stay format-stable
  across engines; row presence is proven by the full scan.
