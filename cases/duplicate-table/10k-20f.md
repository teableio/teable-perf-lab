# duplicate-table/10k-20f

## Goal

Measure duplicating a 10,000-record mixed 20-field table with records included.

## Seed Phase

Create a deterministic source table in the seeded base with 10,000 records and
20 mixed fields. The field shape mirrors
`record-paste/mixed-10k-20fields-complex-copy-paste`: text, long text, single
select, multiple select, number, date, checkbox, rating, and additional text
columns.

Seed data is generated from the row number so V1, V2, and repeated runs compare
the same values. The runner validates the source table with a paged full scan
before measurement.

## Execute Phase

1. Reuse or create the seed source table.
2. Call `POST /api/base/{baseId}/table/{tableId}/duplicate` with
   `includeRecords: true`, then assert the response routing matches the
   requested V1/V2 engine.
3. Resolve the duplicated field and view ids from the response maps.
4. Full-scan the duplicated table through `getRecords` and verify all 10,000
   records and all 20 fields.
5. Delete the duplicated table during local cleanup. The reusable source table
   remains cacheable.

## Primary Metric

- `duplicateTableRequestMs`: `POST /duplicate` request duration with
  `includeRecords: true`. This includes table structure duplication and record
  value copy until the API returns.

## Verification Metrics

- `duplicateTableFullScanReadyMs`: post-response full-scan verification time.
  This proves all 10,000 duplicated records are readable and correct, but it
  does not participate in the primary threshold.
- `duplicateTableTotalReadyMs`: request duration plus full-scan verification
  time, kept as an end-to-end reference.

## Notes

- Source table creation and seed validation are setup phases and are not part of
  the primary metric.
- The case records duplicate-table routing headers and fails V1/V2 runs on
  engine mismatch, so dispatch can be checked from the artifact.
